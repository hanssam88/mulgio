순서 대신 동시 — Worker 3개가 큐를 분배한다

JS 이벤트 루프 덕분에 Race Condition 없이 처리량이 3배가 됐다.

Before — 순차
호텔A ──처리──▶ 완료 호텔B ──처리──▶ 완료 호텔C ──▶ 총 = A + B + C 시간
After — 병렬
호텔A ──처리──▶ 완료 호텔B ──처리──▶ 완료 (동시) 호텔C ──처리──▶ 완료 총 = max(A, B, C) 시간
Worker 패턴 구조
const HOTEL_CONCURRENCY = 3; const hotelQueue = [...hotels]; // 공유 큐 const workers = Array(3).fill(null).map(async () => { while (hotelQueue.length > 0) { const hotel = hotelQueue.shift(); // race condition 없음 await processHotel(hotel); } }); await Promise.all(workers);
핵심: JS 싱글 스레드 특성상 hotelQueue.shift()는 원자적 — 별도 락 없이 안전한 큐 분배

문제가 생길 때만 멈춘다 — 평소엔 0ms

Proactive sleep을 버리고 429 응답에만 반응하는 것만으로 수십 분을 돌려받았다.

Before — Proactive
// 매 호텔마다 무조건 sleep(500); // hotel delay sleep(200); // candidate delay // = 700ms 항상 대기
After — Reactive
// fast 프리셋 hotelDelayMs: 0 candidateDelayMs: 0 // 429 응답 시에만 await sleep(10_000~30_000)

조회도 안 했는데 왜 기다리나

preMappingList가 비어있으면 대기 자체를 건너뛴다 — 작지만 논리적으로 완결된 개선.

Before
for (const hotel of hotels) { const list = preMappingList; // candidate 조회 if (list.length > 0) { await fetchCandidates(list); } // 항상 대기 (조회 안해도) await sleep(candidateDelayMs); }
After
for (const hotel of hotels) { const list = preMappingList; if (list.length > 0) { await fetchCandidates(list); // 조회한 경우만 대기 await sleep(candidateDelayMs); } // 빈 경우 즉시 다음 호텔 }
효과: 매핑 완료 호텔 비율이 높을수록 수혜 증가 — slow/normal 프리셋에서도 candidateDelayMs 절감 효과 적용됨
