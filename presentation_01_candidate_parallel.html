룸 100개 감사에 왜 20초가 걸렸나

순차 호출 구조에서 I/O 대기 시간은 곧 낭비된 벽시계 시간이다.

순차 실행 구조
룸 1 ──[fetch]──▶ 200ms 대기 룸 2 ──[fetch]──▶ 200ms 대기 룸 3 ──[fetch]──▶ 200ms 대기 ⋮ 룸 N ──[fetch]──▶ 200ms 대기 총 대기 = N × 200ms
실측 낭비
룸 100개 → 약 20초 순수 대기
원인 분석
  • ① preMappingList 순회 시
    getRoomMappingCandidates를 1건씩 호출
  • ② 각 API 호출마다 ~200ms 네트워크 대기 발생
  • ③ 병렬 처리 없이 N개 룸 × 200ms 순차 누적
핵심 통찰
I/O 대기 중 CPU는 놀고 있다.
그 시간에 다음 요청을 보내면 된다.

3가지 개선 전략

동시성 제한(concurrentMap)으로 안전하게 3배를 만든다.

  • 01
    동시 3개 병렬 처리 (concurrentMap)
    → 처리량 ~3배 향상 · Rate Limit 안전한 동시성
  • 02
    concurrentMap 유틸 분리 (auditCandidateFetcher.ts)
    → 재사용 가능한 동시성 제어 모듈 — Trip.com API 기준값 확립
  • 03
    TDD 기반 구현
    → 회귀 없이 10개 신규 테스트 추가 (22 → 32개)

concurrentMap(list, 3, fetchFn) — Before / After

for 루프 한 줄이 concurrentMap 한 줄로 교체됐다.

Before — 순차
for (const room of preMappingList) { // 1건씩 순차 호출 const res = await getCandidate(room); results.push(res); // 룸A → 룸B → 룸C ... }
After — 병렬 3개
const results = await concurrentMap( preMappingList, 3, // 동시 최대 3개 getCandidate ); // 룸A, 룸B, 룸C 동시 처리
Before — 순차 흐름
룸A ──200ms──▶ 룸B ──200ms──▶ 룸C ──200ms──▶ 총 = 3 × 200ms = 600ms
After — 병렬 흐름
룸A ──200ms──▶ 룸B ──200ms──▶ (동시) 룸C ──200ms──▶ 총 = ceil(3/3) × 200ms = 200ms
원리: JS 이벤트 루프 + Promise 병렬 실행 — 동시 N개를 초과하지 않으면서 큐를 순차 소비

수치가 구조의 선택이 옳았음을 증명한다

속도만이 아니라 안정성과 품질도 함께 개선됐다.
